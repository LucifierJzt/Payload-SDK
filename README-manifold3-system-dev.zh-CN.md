# 妙算系统开发说明

这份说明面向“我要基于妙算 3 做系统侧开发”的场景。

先把边界说清楚：

- `Payload-SDK` 不是妙算系统源码。
- 这个仓库也不能拿来编译妙算固件或 Linux 发行版。
- 这个仓库提供的是“运行在妙算上的 PSDK 应用开发能力”。

所以如果你说的“开发妙算系统”实际是下面两类之一，需要分开处理。

## 1. 两种完全不同的开发目标

### 目标 A：开发跑在妙算上的业务程序

这是最常见的需求，比如：

- 相机/云台/负载控制
- AI 推理
- 图像处理
- 数据回传
- 与飞机、遥控器、Pilot 交互

这种场景里：

- 妙算 3 提供 Linux 运行环境
- 你的程序跑在妙算用户态
- 如果要和飞机系统深度对接，就接入 PSDK

这时 `Payload-SDK` 仓库是有用的。

### 目标 B：开发妙算系统本身

比如你真想做的是：

- 改系统镜像
- 改系统服务
- 改驱动
- 改启动流程
- 改板级网络、USB、设备树、内核或 rootfs

那这已经不是 PSDK 仓库的工作范围了。

这类事情通常依赖：

- DJI 官方是否开放对应系统能力
- 妙算设备镜像/系统权限
- 设备侧 root 权限和可维护性约束

`Payload-SDK` 本身不提供这些系统源码或构建链路。

## 2. 如果你的真实目标是“在妙算系统上开发应用”

那你应该把工作拆成两层：

### 系统层

系统层负责：

- Linux 运行环境
- 网络
- 存储
- 进程管理
- 日志
- 开机启动
- 设备访问权限

### PSDK 层

PSDK 层负责：

- 飞机通信
- 负载身份
- 负载服务启动
- Pilot 侧展示和交互
- `.dpk` 应用安装包

一句话说：

妙算是“宿主系统”，PSDK 是“接入飞机生态的 SDK”。

## 3. “一盒启动”要看你走哪条路径

这里也要分开看。

### 路径 A：你开发的是 PSDK 应用

建议启动方式：

- 编译业务程序
- 用 `app.json` 打包成 `.dpk`
- 安装到妙算 3
- 交给 DJI 应用管理框架启动

这条路径更适合产品化交付。

### 路径 B：你开发的是普通 Linux 系统应用

建议启动方式：

- 把程序部署到妙算系统
- 用 `systemd` 或系统启动脚本托管
- 自己负责日志、重启、依赖顺序和升级

这条路径更像标准嵌入式 Linux 开发。

## 4. 你现在这个仓库能做什么

当前仓库能直接帮你做的是：

1. 让你的程序在妙算 3 上跑起来。
2. 让程序通过 PSDK 和飞机/云台/负载链路通信。
3. 让程序被打包成 `.dpk` 安装包。
4. 让程序作为“妙算上的 PSDK 应用”被系统识别和管理。

当前仓库不能直接帮你做的是：

1. 重编妙算系统镜像。
2. 改妙算底层 Linux 发行版。
3. 改驱动/内核/boot 流程。
4. 管理系统级 rootfs 打包发布。

## 5. 如果你是做“妙算应用开发”，建议的工程结构

更合理的理解方式是：

- `Payload-SDK` 负责飞机接入能力
- 你的业务代码单独放在自己的应用目录
- 最后把业务代码和 PSDK 集成

推荐结构可以是：

```text
your-project/
├── app/
│   ├── main.cpp
│   ├── inference/
│   ├── pipeline/
│   └── services/
├── config/
├── scripts/
├── third_party/
└── vendor/
    └── Payload-SDK/
```

如果你后面打算做的是长期项目，不建议直接把所有业务逻辑硬塞进官方 sample。

## 6. 什么时候该用 `.dpk`

满足下面任意几条，就应该优先考虑 `.dpk`：

- 你要接入 DJI 应用管理
- 你要在 Pilot 里看到应用信息
- 你要做版本管理
- 你要走安装包分发
- 你要让应用以 PSDK 正式形态运行

## 7. 什么时候该用 `systemd`

满足下面情况时，`systemd` 更像正确方案：

- 你只是做普通 Linux 后台服务
- 服务不依赖 DJI 应用托管机制
- 你更关心系统守护、依赖顺序和自动拉起
- 这个进程不是要作为 DJI 应用包对外安装

但如果这个进程同时又是 PSDK 正式应用，就不要优先走 `systemd`。

## 8. 对你当前目标的直接建议

如果你说“我是准备开发妙算系统”，我建议你先明确自己究竟是哪一种：

### 方向 1：妙算上的 Linux 应用开发

那我建议下一步做：

1. 先把本地工具链装好。
2. 先编通一个普通 aarch64 Linux 程序。
3. 再决定这个程序是否需要接 PSDK。
4. 如果需要接飞机生态，再接入这个仓库。

### 方向 2：妙算上的 PSDK 应用开发

那我建议下一步做：

1. 先改 `dji_sdk_app_info.h`。
2. 先改 `app.json`。
3. 先编出第一个 `.dpk`。
4. 先打通安装和启动链路。

### 方向 3：妙算系统底层定制

那你现在这个仓库不是主战场。
你需要先确认 DJI 是否提供：

- 系统权限
- 固件定制能力
- 底层接口文档
- 系统镜像交付机制

## 10. 一句话结论

如果你要开发的是“妙算系统本身”，`Payload-SDK` 不够。

如果你要开发的是“跑在妙算系统上的飞机负载应用”，那 `Payload-SDK` 是对的，而且正式启动路径优先是：

`应用代码 -> PSDK 集成 -> .dpk -> 安装到妙算 3 -> 应用管理框架启动`
